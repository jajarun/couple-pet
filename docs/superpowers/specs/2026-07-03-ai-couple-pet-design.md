# AI 情侣分身宠物游戏 · 设计文档

- **日期**：2026-07-03
- **状态**：设计已确认，待进入实现计划
- **一句话**：一款给情侣双人配对玩的手机网页游戏——**你的"宠物"就是你对象的 AI 分身**。你设定"对方眼里的你"，你饲养"你眼里的对方"；对一方做的操作（骂、喂狗粮、想你…）会隔空传到另一方，由 AI 分身即时沙雕反应 + 真人本尊亲自回应，主打整蛊无厘头、辅以甜宠，全程 DeepSeek 驱动。

> **说明**：本设计**取代**先前的《AI 沙雕宠物养成游戏》（单机版）方向。单机版的后端地基（账号/JWT、FastAPI 骨架、MySQL、DeepSeek 可替换接口、纯函数规则引擎 + 时间戳懒计算）在 `feat/backend-foundation` 分支，**大部分可复用**。

---

## 1. 愿景与核心体验

- **主打**：情侣双人、整蛊无厘头搞笑、AI 驱动，画面不求精致。
- **最有意思的点**：**双向隔空互动**——"我对 TA 做了个操作，TA 真的收到了、AI 分身即时回怼、TA 本人还能亲自回应我。"
- **情感钩子**：
  - **短期**——分身被 AI 整得又贱又好笑，图一乐。
  - **中期**——"TA 真的在乎/在闹我"（本尊回应的心动峰值）+ 分身按你俩的相处慢慢变异。
  - **长期**——养出一只"你俩专属、独一无二"的分身，关系数值成为你俩共同的牵挂。
- **最大风险**：不是技术，是"隔空互动到底好不好玩、AI 够不够沙雕、会不会玩着玩着真吵架"。第一版首要目标是**做出可双人试玩的成品验证乐趣**。

## 2. 核心设计决策（骨架）

| 维度 | 决策 |
|---|---|
| 关系结构 | 情侣**双人配对**，每对 2 只镜像分身 |
| 宠物本质 | **你的宠物 = 对方的 AI 分身** |
| 反应驱动 | **混合**：AI 分身即时兜底 + 真人本尊回应给情感峰值 |
| 分身人设 | **双方共塑**：本尊设基础人设 + AI 补全 + 饲主互动养成 |
| 互动桥 | **隔空事件流 + 共享关系数值**（两者都要） |
| 触达方式 | **App 内事件流优先**；Web Push 后置，留好接口 |
| 情绪光谱 | **整蛊为核心 + 甜宠为安全阀** |
| 共享数值 | **一套共享池**（两人看同一组数字） |
| 内容尺度 | **打情骂俏放行，露骨硬拦** |
| 实时方案 | **轮询事件流**（回合制场景够用，留好升级 SSE 的位） |
| 技术栈 | **React（Vite）+ FastAPI + MySQL + DeepSeek** |
| AI 成本 | 每用户**每日 AI 反应软上限** + 便宜动作走本地模板 |

## 3. 系统架构与拓扑

### 对称拓扑（谁是谁的宠物）
一对情侣 = 一条**配对关系**，内含 **2 只镜像分身**：
- 女方屏幕上的宠物 = **男友分身**；男方屏幕上的宠物 = **女友分身**。

每只分身绑两个角色：
- **本尊（subject）**：它代表的真人，**设定基础人设**、可随时"附身"真人回应。
- **饲主（keeper）**：天天跟它互动的另一半。

> 一句话：**你设定的是"对方眼里的你"，你养的是"你眼里的对方"。**

### 一次互动的数据流（女方骂男友分身）
1. **AI 即时兜底**：后端用男友分身人设现编回怼 → 立即显示在女方屏幕。
2. **写事件 + 路由真人**：生成一条 `event`（"她骂了你：'大猪蹄子'"）投进**男方本人**的事件流。
3. **真人回应（可选）**：男方本人回怼/服软/甩动作 → 回应事件路由回女方 → 女方那只分身带"👤 本尊回应"标记。
4. **共享数值更新**：委屈值/狗粮值/想你值随之变化。

### 四个部分
- **React 前端**：移动端竖屏，两边镜像。定时轮询 `GET /events?since=<游标>` 拉新事件 + 共享数值。规则逻辑全在后端，前端纯展示。
- **FastAPI 后端**：账号、配对、分身/人设管理、互动动作、事件流、共享数值结算、DeepSeek 代理。
- **MySQL**：用户、配对、分身、共享数值、事件。
- **DeepSeek**：分身反应、人设塑造、无厘头文案，只在后端调，key 藏服务端。

### 从旧地基复用
账号/JWT、FastAPI 骨架、MySQL/SQLAlchemy、DeepSeek Stub 可替换接口、"纯函数规则引擎 + 时间戳懒计算"模式。旧的单机养成模型（孵蛋/图鉴/任务）大部分被本模型替换。

## 4. 数据模型（MySQL）

原则：AI 驱动/会长大的数据用 **JSON 字段**；随时间变化的关系数值用**时间戳懒计算**（不跑定时任务）。

### `users`（账号）
`id` · `nickname`(唯一) · `password_hash` · `ai_count`(每日 AI 额度) · `ai_count_date` · `created_at` · `last_login_at`

### `couples`（配对关系）
`id` · `user_a_id` · `user_b_id`(配对前为空) · `pair_code`(邀请码) · `status`(`pending`/`active`) · `created_at` · `paired_at`
> 配对：一方创建 → 拿 pair_code/链接发给对方 → 对方输入 → 关系变 `active`，同时生成两只分身占位。

### `avatars`（分身，每对 2 行，镜像）
- `id` · `couple_id`
- `subject_user_id`（本尊 = 代表谁、谁定人设、谁能附身）
- `keeper_user_id`（饲主 = 谁天天互动）
- `name` · `appearance`(JSON：emoji+颜色+外形描述)
- `persona`(JSON：本尊设定的基础人设 + 后天长出的特质)
- `evolution`(JSON：互动影响向量，驱动分身变异/进化)
- `memory_summary`(TEXT：AI 压缩历史，控 token)
- `created_at`

### `couple_stats`（共享关系数值，每对 1 行）
`couple_id`(主键) · `stats`(JSON：委屈值/狗粮值/想你值/亲密度…) · `stats_updated_at`
> 懒计算：想你值随多久没互动而涨；委屈值随时间/被哄而降。**一套共享池，两人看同一组数字。**

### `events`（核心：情侣共享时间线，append-only）
`id`(主键，**同时是轮询游标**) · `couple_id` · `actor_user_id` · `kind`(`action`/`ai_reaction`/`real_response`/`system`) · `action_type`(scold/feed_dogfood/poke/hug/miss/apologize…) · `content`(TEXT/JSON) · `parent_event_id`(真人回应指向被回应的操作，串成"她骂→他回") · `created_at`
> 轮询：`GET /events?since=<上次最大id>` 一次拿回新事件；两人共享时间线，各按视角渲染。

## 5. 核心互动循环

三块：**捏分身（入口）→ 日常互动（骂+宠走统一数据流）→ 分身成长**。

### 捏分身（配对后各捏各的）
两人各自设定"对方将看到的那只分身"（= 你眼中的你）：选基调（毒舌/憨憨/舔狗/高冷/中二…）+ 名字 + emoji 造型 + 答几个种子问题，AI 扩写成完整人设。这就是"你亲手决定对方眼里的你是个什么沙雕"。

### 日常互动（每个动作走统一数据流）
所有动作 = **① AI 即时用分身人设反应（给你看）→ ② 生成事件路由给真人对方 → ③ 对方可"本尊附身"真人回应 → ④ 共享数值变化**。

**整蛊类（核心）**：骂一顿（打字/模板）、戳一戳/弹脑门/拧耳朵、查岗、吐槽/告状。
**甜宠类（安全阀）**：喂狗粮（狗粮值↑、委屈值↓）、抱抱/想你（想你值→亲密度）、道歉·和好（委屈值↓、亲密度↑）。
**随便唠**：输入框跟分身打字聊天，AI 用人设回，同时作为事件同步给对方。（聊天是养灵魂和无厘头笑点主战场）

### 共享数值的情绪节拍
- 骂/整蛊 → **委屈值↑**（玩笑式）；喂狗粮/甜 → **狗粮值↑、委屈值↓**；长时间不互动 → **想你值↑**。
- **委屈值过高 → AI 旁白 + 分身闹脾气提示"该哄了"——安全阀**，把互怼自然导向和好，避免玩笑玩成真吵架。

### 分身成长
`evolution` 影响向量记录饲主的相处方式 + 本尊回应风格。满足条件触发**进化事件**（AI 改写人设 + 沙雕过场）：被骂多了觉醒杠精、被宠多了变粘人精、长期冷落离家出走…既是无厘头高光，也是"独一无二"的情感回报。

### 每日循环
打开 → 看对方分身 + 共享数值 + 事件流（你不在时对方对你干了啥，配 AI 反应）→ 处理待回应事件（本尊附身回怼/服软或纯看乐）→ 对 TA 做动作 → 对方收到事件 → 分身慢慢长 → 委屈值/想你值推着你俩互动。

**防刷/控成本**：每用户**每日 AI 反应额度** + 动作**冷却**（避免刷屏轰炸对方、护 API 钱包）。

## 6. AI / DeepSeek 集成

### DeepSeek 的活
| 活 | 频率 | 成本 |
|---|---|---|
| 分身人设生成/扩写 | 低 | 中 |
| **分身即时反应**（动作+聊天） | **高** | 主要开销 |
| 进化事件文案 | 低 | 中 |
| 系统旁白（委屈值爆表调解） | 偶尔 | 低 |
| 记忆压缩 | 低 | 低 |

便宜动作（戳、抱抱）走**本地模板+随机**，不烧 API；只有骂/聊天/关键时刻才叫 DeepSeek。

### Prompt 架构
每次现拼：`分身人设 + 当前共享数值 + 本次操作内容 + 记忆摘要 + 安全护栏` → 回复短、in-persona、无厘头。`deepseek-chat` 模型，`max_tokens` 压短，滚动记忆控上下文，每用户每日额度封顶。

### 🔑 情侣产品特有安全护栏
- **AI 分身是"搞笑缓冲垫"**：饲主打了很伤人的话，分身**不复读、不升级**，而是**把攻击性化解成无厘头搞笑**。system prompt 写死"绝不人身攻击真人、绝不升级真实冲突，一律往沙雕化解"。
- **内容红线**：仇恨/威胁/暴力/自残一律拦截并温和岔开；辱骂保持"打情骂俏"程度。玩家输入加轻过滤。
- **亲密尺度**：打情骂俏/暧昧擦边放行，**明确露骨/性描写硬拦**。
- **委屈值安全阀**：委屈值过高 AI 主动旁白引导去哄。
- **真人回应**：保留原味；对真人回应的内容审核留作升级位。

### 失败兜底
API 超时/报错 → 符合人设的卖萌文案顶上 + 退避重试；**失败不改数值**；不把报错糊玩家脸上。

## 7. 界面与交互

移动端竖屏，两边镜像。emoji + 小动画 + 大字号沙雕文案 + 情侣软萌撑氛围。

### 屏幕结构（底部三 tab：🏠 TA / 🔔 事件流 / ⚙️ 我的分身）
- **🏠 家 / TA 的分身（核心）**：顶部共享数值仪表盘（委屈/狗粮/想你/亲密度）；中间对方分身本体（idle 动画）+ 头顶气泡；底部两排按钮（整蛊类 + 甜宠类）+ "💬 随便唠"。
- **🔔 事件流（时间线）**：你俩共享时间线，"她骂→他回"串成串，在此**本尊附身回应**。**对方本人真回应的那条带醒目"👤 本尊回应"标记 + 特殊动效**——心动峰值靠它，重点强化。新事件轮询到轻"叮"一下。
- **⚙️ 我的分身**：管理你自己那只（对方看到的你）——人设、名字、造型。
- **💬 聊天**：从家滑出，跟对方分身打字聊，打字机流式，同步为事件。
- **⚡ 进化事件**：全屏浮层演夸张过场。

### 新手引导
注册 → 创建情侣（拿邀请码/链接发 TA）或输入对方邀请码加入 → 配对成功 → 捏你自己的分身 → 落到家界面。
> **等待态**：对方没配对/没捏好分身时，家界面显示"对方分身孵化中 + 催 TA 一下"占位，别对着空屏。

### 视觉基调
**复古电子宠物机 + 表情包 + 情侣软萌**：干净、俏皮、大字号、meme 感。可在实现阶段微调。

## 8. 错误处理与测试

### 错误处理
- **AI 失败**：见 §6 兜底。
- **配对边界**：邀请码无效/过期、重复配对、自配、对方解绑离开——温和处理 + 清晰提示。
- **动作幂等**：一个操作只产生一条事件，网络重试**不重复轰炸对方**。用客户端幂等键。
- **轮询游标**：`events.id` 单调递增，`since=<id>` 只返回更新事件，不漏不重。
- **共享数值并发**：`couple_stats` 更新走**事务 + 原子增减**，防两人同时改互相覆盖。
- **权限校验**：用户只能在自己配对关系内操作、只能动自己饲养的分身、只能回应发给自己的事件——绝不能戳陌生人分身。
- **等待态**：对方没上线 → 事件在流里等，界面给占位与"催 TA"。
- **账号**：JWT 过期静默刷新或温和引导重登。

### 测试策略
- **后端（重点）**：
  - **纯函数规则引擎单测**：共享数值增减/懒衰减、进化触发、委屈值安全阀阈值。
  - **配对逻辑**：创建/加入/无效码/重复配对/自配。
  - **互动流**：动作→一条事件→对方轮询拿到→真人回应→路由回来；**幂等**（重复动作键只产生一条事件）。
  - **权限**：用户 B 不能操作 C 的关系、不能回应不属于自己的事件。
  - **DeepSeek 全程 Mock**，CI 不调真 API；测 prompt 拼装 + 坏输出降级 + 安全护栏化解逻辑。
- **前端**：关键流程（配对、捏分身、动作按钮、事件流、本尊回应）组件测试，后端 Mock。
- **真人试玩**：好不好玩只能靠**两台设备/两个账号真配对玩一遍**，单人测不出隔空互动的爽点。

## 9. 非目标（本版明确不做，YAGNI）

- Web Push 推送（留好接口，后置）。
- SSE/WebSocket 实时（先轮询，数据模型留好升级位）。
- 分方向的关系数值账（先一套共享池）。
- 第三方登录、多对关系、群体社交。
- 露骨/开车内容。
- 真人回应的内容审核（留升级位）。
- 精致美术。

## 10. 升级路径（留好的位）

- 事件送达层从轮询升级到 SSE（数据模型不动）。
- Web Push 触达对方手机。
- 关系数值细账 / 年度报告 / "谁更舔"排行。
- 真人回应内容审核。
- 图鉴式回顾你俩养过的分身进化史。
